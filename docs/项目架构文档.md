# 竞彩预测网页项目架构文档

## 1. 架构概述

### 1.1 设计理念

本项目采用流式处理架构，以Python全栈为核心，实现从数据采集到预测展示的完整流程。系统设计注重开发效率和代码简洁性，通过单一语言和模块化设计，确保Vibe Coding的核心优势。

| ------ | --------------------- | ------- | ------------------------ |

| 后端     | FastAPI               | 0.104.x | 高性能异步Web框架，自动生成API文档     |
| 数据库    | SQLite (初期) → DuckDB  | 最新版     | 零配置，数据分析友好，完美契合Pandas数据流 |

| 数据可视化  | Plotly                | 5.18.x  | 交互式图表，与Streamlit完美集成     |

## 3. 数据流设计

### 3.1 数据采集流程

1. 数据采集脚本定期从公开API获取历史比赛数据
2. 数据存储到PostgreSQL数据库
3. 数据清洗模块处理数据质量问题
4. 因子计算模块基于清洗后的数据计算各项因子

### 3.2 模型训练流程

1. 从数据库提取历史比赛数据和计算好的因子
2. 按时间序列划分训练集和验证集
3. 训练胜负预测模型和进球数预测模型
4. 评估模型性能，保存最优模型

### 3.3 预测流程

1. 前端用户选择要预测的比赛
2. 后端API接收请求，查询相关历史数据
3. 计算比赛的各项因子值
4. 加载训练好的模型进行预测
5. 计算胜负概率、最可能比分等结果
6. 将预测结果返回给前端展示

## 4. 核心功能实现

### 4.1 数据处理模块

#### 4.1.1 数据采集

- **实现方式**：Python脚本定时运行，调用第三方足球数据API
- **数据范围**：覆盖英超、西甲、德甲等主流联赛近5个赛季的比赛数据
- **更新频率**：每日更新，确保数据时效性

#### 4.1.2 因子计算

- **实现方式**：Python函数库，支持多种因子计算
- **核心因子**：
  - 进攻端因子：场均预期进球、射正次数等
  - 防守端因子：场均预期失球、被射正次数等
  - 交互因子：A队进攻与B队防守的对比
  - 状态趋势因子：加权近期数据、实际积分vs预期积分
  - 交锋历史因子：双方近K次交锋数据
  - 赛程与情景因子：休息天数差、主客场标识等
  - 统计离散与稳定性因子：预期进球/失球的方差
  - 赛季绝对实力与底蕴因子：排名差值、历史排名等
  - 赛季攻防统治力因子：净胜球差值、胜率差值等
  - 对阵不同档位球队的表现因子：对阵上下半区球队的表现
  - 当前赛季战意/压力因子：与欧战区/降级区的分差
  - 核心球员缺阵影响因子：建立一个核心球员缺阵记录，与历史比赛关联。对“A队某射手伤缺时，其他球员实际进球与预期进球的差值”进行量化。
  - 球队疲劳指数：使用赛前休息天数、连续跑动数据等，构建一个0-1的疲劳度因子。

### 4.2 模型训练模块

#### 4.2.1 胜负预测模型

- **实现方式**：
  - 逻辑回归：用于基线模型和可解释性
  - XGBoost：用于提高预测准确率
- **训练方法**：时间序列交叉验证，确保模型泛化能力
- **评估指标**：准确率、F1-score、对数损失

#### 4.2.2 进球数预测模型

- **实现方式**：
  - 双泊松分布：传统方法，适合计数数据
  - XGBoost回归：现代方法，捕捉非线性关系
- **训练方法**：使用Poisson或Tweedie损失函数
- **评估指标**：均方误差、对数似然

### 4.3 预测服务模块

#### 4.3.1 API设计

- **RESTful API**：提供比赛预测、数据查询等接口
- **主要端点**：
  - `/api/predict`：预测比赛结果
  - `/api/teams`：获取球队列表
  - `/api/matches`：获取比赛列表
  - `/api/factors`：获取因子详情

#### 4.3.2 性能优化

- **缓存机制**：使用Redis缓存频繁查询的数据
- **异步处理**：使用Node.js异步IO处理并发请求
- **模型加载**：预加载模型到内存，减少预测延迟

### 4.4 前端展示模块

#### 4.4.1 页面结构

- **比赛选择页**：搜索框、热门比赛列表
- **预测结果页**：胜负概率饼图、比分预测表、总进球数分布
- **数据分析页**：因子分析、历史验证、模型表现

#### 4.4.2 交互设计

- **实时更新**：选择比赛后自动更新预测结果
- **数据筛选**：允许用户调整数据时间窗口
- **详情展开**：点击因子可查看详细计算过程
- **结果导出**：支持导出预测结果为PDF或Excel

## 7. 性能优化策略

### 7.1 数据处理优化

- **批量处理**：使用批量操作减少数据库查询次数
- **并行计算**：使用多线程并行计算因子
- **缓存机制**：缓存计算结果，避免重复计算

### 7.2 模型优化

- **特征选择**：使用特征重要性分析，选择最相关的因子
- **模型压缩**：使用模型压缩技术，减少模型大小
- **增量学习**：支持增量学习，适应数据变化

### 7.3 服务优化

- **负载均衡**：使用负载均衡器分散请求压力
- **水平扩展**：根据流量自动扩展服务实例
- **响应缓存**：缓存热门比赛的预测结果

## 8. 安全性考虑

### 8.1 数据安全

- **数据加密**：加密存储敏感数据
- **访问控制**：实现基于角色的访问控制
- **数据备份**：定期备份数据库，确保数据安全

### 8.2 API安全

- **认证授权**：使用JWT进行API认证
- \*\* rate limiting\*\*：实现API请求限流，防止滥用
- **输入验证**：严格验证API输入，防止注入攻击

### 8.3 系统安全

- **漏洞扫描**：定期进行安全漏洞扫描
- **更新依赖**：及时更新依赖库，修复安全漏洞
- **日志监控**：监控系统日志，及时发现异常

## 10. 结论

本架构设计基于分层架构理念，将系统分为数据处理、模型训练、预测服务和前端展示四个核心层次，确保了系统的可扩展性、可维护性和高性能。通过合理的模块划分和数据流设计，系统能够高效处理历史比赛数据，训练准确的预测模型，并为用户提供直观、专业的预测结果。

架构设计充分考虑了系统的性能优化、安全性和扩展性，为后续的功能扩展和技术升级预留了空间。通过持续的模型优化和数据更新，系统将不断提升预测准确率，为用户提供更有价值的竞彩预测服务。
